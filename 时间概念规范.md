## 📅 核心时间概念定义

### 1. 自然日
**定义**：日历上的每一天（00:00-23:59）  
**业务含义**：质检日报文件发出日，每天都会有质检日报发出  
**在项目中的运用**：
- 日常操作的时间基准
- 出入池操作的执行日
- 每日报告生成的日期

**关键规则**：自然日 = 质检日报文件发出日

### 2. 质检日报文件发出日
**定义**：质检部门正式发布日报文件的日期  
**业务含义**：质检员完成数据整理并发布日报的日期  
**命名规范**：`MMDD技术支持二中心 质检日报.xlsx`
- 示例：`0328技术支持二中心 质检日报.xlsx`
- 含义：反映3月28日产生的工单质量

**文件格式**：Excel文件

### 3. 工单抽检日
**定义**：质检人员实际抽检工单的日期  
**业务含义**：质检员执行抽检工作的日期

### 4. 工单产生日
**定义**：客服人员处理工单的实际日期  
**业务含义**：技术支持团队实际处理工单的日期

## 🔄 时间关系规则

### 核心业务规则
**"今天抽昨天产生的工单，明天发今天抽检的数据"**

### 时间链条
```
工单产生日 → (+1天) → 工单抽检日 → (+1天) → 质检日报文件发出日（=自然日）
```

### 具体示例
| 环节 | 日期 | 说明 |
|------|------|------|
| 工单产生日 | 3月28日 | 技术支持团队处理工单 |
| 工单抽检日 | 3月29日 | 质检员张三抽检28日的工单 |
| 质检日报文件发出日 | 3月30日 | 张三发出日报，命名为"0328" |
| 自然日 | 3月30日 | 日常操作日期，与日报发出日相同 |

### 计算公式
```python
# 已知：日报文件名中的日期为工单产生日
work_order_date = parse_date_from_filename(filename)  # 如"0328"→"2026-03-28"
check_date = work_order_date + timedelta(days=1)      # 工单抽检日
report_date = check_date + timedelta(days=1)          # 质检日报文件发出日
natural_day = report_date                             # 自然日
```

## ⚠️ 异常情况处理

### 1. 节假日延迟
**场景**：质检员休息日不发日报，延迟到上班后发送  
**示例**：
- 4月5日清明节休息
- 原本5号应该发"0403技术支持二中心 质检日报"
- 实际6号发"0403技术支持二中心 质检日报"

**处理规则**：
- 日报命名不变（仍反映工单产生日）
- 日报发出日顺延
- 时间计算逻辑不变

### 2. 长假期合并
**场景**：长假期后合并发送多日数据  
**示例**：
- 春节2月14日-2月21日休息
- 2月22日上班后，发送"【0214-0222】技术支持二中心 质检日报"

**命名格式**：`【起始日期-结束日期】技术支持二中心 质检日报.xlsx`

**处理规则**：
- 需要特殊解析合并日报
- 数据覆盖多个工单产生日
- 需要人工确认处理方式

### 3. 数据错误处理
**扣罚错误**：通过"申诉窗口"处理  
**其他错误**：暂时不考虑，遇到时特殊处理

## 🎯 在质检训练营项目中的运用

### 1. 培训效果评估
- **评估依据**：工单产生日的表现
- **数据获取**：质检日报文件发出日
- **时间延迟**：2天（工单产生日 → 日报发出日）

### 2. 出入池决策
- **决策时间**：自然日（日报发出日）
- **决策依据**：最新日报数据
- **时间考量**：考虑2天数据延迟的影响

### 3. 数据关联逻辑
```python
# 人员培训效果评估
def evaluate_training_effect(staff_id, training_date):
    # 1. 找到培训后处理的工单（工单产生日）
    work_orders = get_work_orders_after_training(staff_id, training_date)
    
    # 2. 获取这些工单的抽检结果（工单抽检日）
    check_results = get_check_results_for_work_orders(work_orders)
    
    # 3. 从日报中获取成绩（质检日报文件发出日）
    daily_reports = get_daily_reports_for_dates(check_results.dates)
    
    # 4. 在自然日进行评估和决策
    evaluation_date = get_current_natural_day()
    return make_decision(staff_id, daily_reports, evaluation_date)
```

### 4. 自动化流程设计要点
1. **时间计算准确性**：必须正确处理+1天的关系
2. **异常处理完备性**：必须支持节假日和合并日报
3. **数据延迟考虑**：决策时要考虑2天数据延迟
4. **容错机制**：命名解析要有容错能力

## 📋 实施注意事项

### 技术实现
1. **日期解析**：从文件名准确提取工单产生日
2. **时间计算**：正确处理跨月、跨年的日期计算
3. **节假日处理**：维护节假日日历，支持特殊规则

### 业务理解
1. **命名含义**：日报命名反映的是工单产生日，不是抽检日或发布日
2. **时间延迟**：理解2天延迟对决策的影响
3. **异常认知**：接受节假日导致的日报延迟是正常现象

### 系统设计
1. **灵活性**：设计可配置的时间规则
2. **可追溯性**：记录完整的时间链条
3. **可审计性**：所有时间计算都要有日志记录

## 🔍 验证规则

### 正确性验证
1. **时间关系验证**：工单抽检日 = 工单产生日 + 1天
2. **日报时间验证**：质检日报文件发出日 = 工单抽检日 + 1天
3. **命名验证**：日报文件名中的日期 = 工单产生日

### 异常验证
1. **节假日验证**：延迟发布的日报命名是否正确
2. **合并日报验证**：特殊命名格式是否能正确解析
3. **边界验证**：月底、年底的时间计算是否正确
